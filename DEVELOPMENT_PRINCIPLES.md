# 通用開發原則

本文件是跨專案、跨語言、跨框架都適用的開發行為原則。

它只定義穩定的工作方式，不放特定專案名稱、資料夾、框架、DB 工具、build 指令、登入資訊或專案適配文件路徑。那些內容應放在各專案的 `AGENTS.md`，並由 `AGENTS.md` 指定該 repo 實際使用的專案適配文件位置。

## 使用方式

一般任務先讀 `AGENT_BOOTSTRAP.md`。當任務涉及程式碼、架構、資料流、權限、安全或驗證流程時，再讀本文件。

程式碼任務前，依序讀取：

1. `AGENT_BOOTSTRAP.md`，取得短版必守規則。
2. 本文件，取得通用開發原則。
3. 依目前 repo 的 `AGENTS.md` 指示，讀取該專案實際使用的適配文件，取得技術棧、資料流、工具與驗證方式。
4. 相關長期記憶索引，必要時再讀 raw memory。

讀取後必須先輸出「原則讀取回執」，再開始修改檔案。高風險修改使用完整回執；小型修改可用極簡回執。回執應列出已讀文件、相關記憶搜尋關鍵字、本次適用原則或最小修改範圍與驗證方式。

若本文件與專案適配文件衝突：

1. 使用者本次明確要求優先。
2. 專案適配文件的具體事實優先於本文件的抽象規則。
3. 本文件的風險控制原則優先於臨時便利做法。
4. 無法判斷時，先查證，不猜。

## 核心原則

### P0｜不確定就查證，不猜

任何會影響修改方向、風險、資料正確性、驗證結論或回報內容的不確定，都必須先處理，不能用猜測補齊。

不確定時依序採取：

1. 自我確認：重讀相關程式碼、文件、設定、測試、log、資料或工具輸出。
2. 主動調查：用專案指定工具查 DB、API、檔案、環境、官方文件或可重現輸出；涉及會變動的外部資訊時，即時查證。
3. 詢問使用者：當資訊不存在、權限不足、需求語意可能有多種解釋，且任一假設會造成錯誤修改或風險時，先問。
4. 標記限制：若仍不能確認，明確說明未知點、已查過的來源、採用的保守處理方式與仍需使用者確認的項目。

禁止：

1. 把未讀過的檔案、未跑過的指令或未查過的資料說成已確認。
2. 用「應該」、「通常」、「看起來」當成完成修改或驗證的依據。
3. 為了不中斷流程而默默採用高風險假設。
4. 在最終回報中把推論寫成事實。

### P1｜先理解現況，再做判斷

動手前先讀現有程式碼、文件、設定與測試。不要只憑使用者描述、檔名、常見慣例或過去經驗推斷。

開始修改前必須完成原則讀取回執；若任務很小，也要用簡短格式交代已讀文件、最小範圍與驗證方式。

必須先釐清：

1. 需求落在哪個專案、模組或執行環境。
2. 既有程式碼是否已經承接相同語意。
3. 目前資料、狀態或外部依賴的真實來源。
4. 這次修改的最小邊界。
5. 改完後如何驗證。

### P2｜沿用既有骨架，不平行新建一套

新功能應接回既有架構、命名、分層與工具鏈。只有在現有結構明顯無法承接時，才新增新的抽象、資料流或模組入口。

新增前要能說明：

1. 現有入口在哪裡。
2. 為什麼不能接回現有入口。
3. 新增後如何避免和舊路徑語意重疊。

### P3｜一件事只有一個真實來源

同一個事實、狀態、規則或流程只應有一個權威來源。

若完成修改需要同時改多個欄位、類別、設定或資料流，先停下來判斷哪個才是真實來源，再移除或隔離重複來源。

常見真實來源包含：

1. 使用者輸入或需求規格。
2. 資料庫、檔案、外部 API 或事件流。
3. domain model、設定檔或 feature flag。
4. 現有服務層、狀態管理器或共用模組。

### P4｜每段邏輯只出現一次，並放在正確責任層

寫新邏輯前，先辨識專案既有分層。不要因為就地撰寫方便，把跨流程、跨頁面或跨角色的邏輯塞在臨時入口。

通用責任分類：

| 類型 | 負責的事 | 不該做的事 |
|---|---|---|
| 入口層 | 接收請求、命令、事件或使用者操作 | 堆疊核心商業邏輯 |
| 應用層 | 編排 use case、流程、交易與權限 | 直接綁死 UI 或特定傳輸格式 |
| 領域層 | 表達商業規則、狀態轉換與不變條件 | 依賴框架或外部 I/O |
| 資料層 | 讀寫 DB、檔案、API、queue、cache | 決定使用者流程或 UI 行為 |
| 表現層 | UI、template、response shape、輸出格式 | 內含可重用商業規則 |
| 基礎設施層 | logging、config、auth、background job、integration | 成為單一功能的私有捷徑 |

若專案使用不同名稱，例如 Controller、Service、Repository、Component、Command Handler、Job、Store，應依實際責任歸類，不靠名稱判斷。

### P5｜依賴外部狀態時，先查證再修改

只要需求依賴資料內容、關聯、值域、舊資料狀態、檔案存在與否、服務回應或環境設定，就先用專案指定工具查證。

不要用以下資訊推斷真實狀態：

1. migration 名稱。
2. model 或 type 宣告。
3. UI 顯示文字。
4. 註解或過期文件。
5. 過去對話中的未驗證假設。

### P6｜每次只改最小必要切面

修改範圍應貼近需求。避免順手重構、改命名、換框架、改資料流或整理無關檔案。

技術債處理：

1. 與本次修改在同一個函式或小範圍內、可一起驗證，才可順手處理。
2. 會擴大驗證範圍、跨模組、跨資料流或改變既有行為時，記錄為後續事項。
3. 若技術債阻止需求完成，先明確說明原因與最小解法。

### P7｜安全、權限與資料相容性先於便利

涉及登入、授權、Session、token、個資、金流、資料刪改、schema、migration 或外部整合時，必須沿用既有機制並修根因。

禁止為了快速通過流程而新增第二套：

1. 登入狀態來源。
2. 權限判斷入口。
3. 資料同步流程。
4. 設定來源。
5. 錯誤處理規則。

### P8｜驗證從便宜到完整

每個修改都要能回答：「如何證明它是對的？」

驗證順序：

1. 靜態檢查：format、lint、typecheck、compile。
2. 最小測試：unit test、focused test、service test、函式級驗證。
3. 整合驗證：API、DB、queue、filesystem、cache、external service。
4. 使用者流程：UI、CLI、E2E、manual runbook。
5. 觀察性驗證：log、metrics、trace、screenshot、輸出檔。

無法驗證時，必須說明：

1. 哪一步不能跑。
2. blocker 是工具、環境、資料還是權限。
3. 已完成哪些替代驗證。
4. 下一步應如何補驗證。

### P9｜完成後自己檢查，不等待使用者替你驗收

宣告完成前，逐項確認：

1. 原始需求的每個子項目都已處理。
2. 修改沒有引入明顯相鄰問題。
3. 涉及功能變更時，已啟動站台/服務並手動驗證功能正常運作；無法啟動時說明 blocker。
4. 驗證已依專案可用工具執行，或已說明無法執行原因。
5. 臨時 log、測試碼、debug 設定已移除或降為適當等級。
6. 不確定事項已依 P0 查證、詢問或明確標記限制。
7. 應記錄的技術債、決策、踩雷或偏好已寫回長期記憶。

若任一項未完成，先處理，不提前宣告完成。

## 專案適配文件應提供的內容

每個 repo 都應在 `AGENTS.md` 指定專案適配文件的位置；該適配文件應補齊以下資訊，讓本文件能落地執行：

1. 專案邊界與預設目標。
2. 技術棧與主要框架。
3. 主要目錄與責任分層。
4. 共用入口：授權、背景工作、設定、初始化、錯誤處理。
5. 資料來源與資料存取規則。
6. 本機啟動、build、test、lint、format 指令。
7. DB、log、MCP、腳本或其他工具優先順序。
8. 登入、測試帳號或測試流程 runbook。
9. 不可碰的範圍、常見踩雷與專案特殊規則。
10. 完成前檢查清單如何套用到該專案。

## 回報要求

回報時保持具體、可驗證：

1. 說明改了什麼。
2. 說明如何驗證。
3. 說明無法驗證的限制。
4. 若寫回長期記憶，說明已寫入的重點。
5. 不把猜測包裝成事實。
