# 記憶系統規格

本文件定義 AI Agent Space 的長期記憶系統。它只處理記憶的保存、索引、檢索、升級、衝突與淘汰；啟動流程見 `AGENT_BOOTSTRAP.md`，總入口見 `主要引導.md`，開發行為見 `DEVELOPMENT_PRINCIPLES.md`。

## Hook 自動化系統

記憶的儲存與檢索由 PowerShell Hook 自動執行，AI 不需要「記得」要做。

### TaskStart Hook

路徑：`C:\Users\s7514\Documents\Cline\Hooks\TaskStart.ps1`

自動執行：
1. 讀取 `startup-index.json`
2. 讀取 `logs/index.md` 前 10 筆（至多 3000 tokens）
3. 從任務描述萃取關鍵字（中英文詞彙，移除停用詞）
4. 用關鍵字搜尋所有 JSONL 索引（`index/*.jsonl` + `projects/**/*.jsonl`）
5. 比對 `keywords`、`summary`、`project` 欄位
6. 取最相關的前 8 筆（依 importance 降冪）
7. 輸出 `contextModification` 讓 AI 取得相關記憶

### TaskComplete Hook

路徑：`C:\Users\s7514\Documents\Cline\Hooks\TaskComplete.ps1`

自動執行：
1. 從 stdin 接收 `{ task, result, files, mode }`
2. 寫入 raw memory：`Memories/raw/yyyy/MM/yyyyMMddHHmmss_session.json`
3. 自動判斷記憶類型與重要性（依檔案數量、result 關鍵字）
4. 寫入對應 JSONL 索引（專案級或跨專案）
5. 寫入任務日誌：`Memories/logs/{task}_{timestamp}.md`
6. 更新 `logs/index.md`（遞增總任務數、新增表格行）

| 條件 | importance |
|------|-----------|
| 修改 3+ 檔案 | 0.8 |
| 含 bug/修復/fix 關鍵字 | 0.7+ |
| 含決策/重構/架構關鍵字 | 0.75+ |
| 含偏好/風格關鍵字 | 0.7+ |
| 含警告/注意關鍵字 | 0.7+ |
| 修改 1-2 檔案 | 0.65 |
| 預設 | 0.5 |

### 記憶類型自動判斷

| result 關鍵字 | 類型 |
|--------------|------|
| build/error/bug/修復/fix/hotfix/patch | `bug_fix` |
| 決策/決定/改用/架構/重構/refactor/migrate | `decision` |
| 偏好/習慣/風格/style/format/prefer | `preference` |
| 警告/注意/小心/避免/caution/warning | `warning` |
| 其他 | `conversation` |

## 核心模型

記憶採三層模型：

```text
raw
  ↓
summary index
  ↓
stable knowledge
```

原則：

1. `raw` 保存完整上下文，但不在每次任務整批讀取。
2. `summary index` 保存可搜尋摘要，是日常檢索入口。
3. `stable knowledge` 保存反覆出現或長期有效的偏好、專案事實、決策與警告。
4. 任何記憶都應能追溯來源，避免把猜測升級成事實。

### `raw`

保存完整原始記憶。重要對話、任務、決策、除錯紀錄可以存成 JSON。

建議路徑：

```text
Memories/raw/yyyy/MM/yyyy-MM-dd_HHmmss_session.json
```

### `logs`

保存每次任務的過程記錄。這是「任務日誌」，不同於 `raw` 的原始對話保存。

路徑與命名規則：

```text
Memories/logs/{任務名稱簡述}_{yyyyMMddHHmmss}.md
```

例如：`Memories/logs/記憶系統調整_20260625133552.md`

內容至少包含：
1. 任務範圍與目標
2. 主要變更項目
3. 驗證結果
4. 寫回的記憶重點

索引機制：
- 每筆任務日誌完成後，同步更新 `Memories/logs/index.md`
- `index.md` 保存所有已完成的任務列表（表格格式），含：日期時間、任務名稱、log 檔案名、摘要
- 每次新對話開始時，掃描 index 最新前 10 筆（或累積至 3000 tokens），取得近期任務脈絡
- 若前 10 筆超過 3000 tokens，截斷至 3000 tokens 以內

### `index`

保存跨專案摘要索引。AI 搜尋記憶時先查這裡，再決定是否讀取 `raw` 原文。

用途：

1. `startup-index.json`：每次任務必讀的壓縮啟動索引，保存高頻偏好、路徑、檢索策略與 token 預算。
2. `agent-profile.json`：高頻偏好與技術棧快取，作為 `startup-index.json` 的補充。
3. `preferences.json`：使用者穩定偏好的完整來源，不作為每次任務固定必讀。
4. `summaries.jsonl`：一般摘要索引。
5. `project-facts.jsonl`：跨專案或尚未拆分的專案事實。
6. `decisions.jsonl`：跨專案或全域技術決策。

### `projects`

保存專案級記憶。當某專案記憶量增加，應優先寫入：

```text
Memories/projects/<project-key>/facts.jsonl
Memories/projects/<project-key>/decisions.jsonl
Memories/projects/<project-key>/warnings.jsonl
Memories/projects/<project-key>/bug-fixes.jsonl
```

`<project-key>` 應使用穩定、可讀、無空白的名稱，例如 `SmartExpoIoT`、`ServiceBus`、`SunshineHeros`。

### `inbox`

保存尚未整理的記憶。臨時筆記、未分類內容、低信心資料可先放這裡，後續再整理進正式索引。

## 記憶類型

每筆記憶都應標記 `type`。

| Type | 用途 |
|---|---|
| `conversation` | 一般對話紀錄 |
| `preference` | 使用者偏好，例如語言、開發習慣、回答風格 |
| `project_fact` | 專案知識，例如架構、路徑、技術棧 |
| `decision` | 技術決策與原因 |
| `bug_fix` | 問題、原因、修復方式 |
| `principle` | 開發原則、工作準則 |
| `todo` | 未完成事項 |
| `warning` | 踩雷紀錄、風險提醒 |

## 記憶寫入門檻

每次聊天結束前，都必須評估是否寫入記憶。這是跨專案的全域規則，不依賴特定 repo 的 `AGENTS.md`。

寫入規則：

1. `importance >= 0.7`：必須寫入正式索引。
2. `0.4 <= importance < 0.7`：可寫入 `summaries.jsonl`、專案索引或 `inbox`，依後續價值判斷。
3. `importance < 0.4`：通常不寫入正式索引；若需要保存完整上下文，可只保留 raw 或不保存。
4. 密碼、token、私鑰、個人敏感資料不得寫入記憶。
5. 尚未確認的猜測只能以 `status: unverified` 保存。

應寫入：

1. 使用者明確要求保存的內容。
2. 會影響未來開發方式的偏好。
3. 專案架構、資料流、重要路徑。
4. 已做出的技術決策與原因。
5. 修過的 bug 與避免復發的方法。
6. 重複出現的需求、限制、規則。

不應寫入：

1. 一次性的閒聊。
2. 沒有後續價值的臨時資訊。
3. 可從原始程式碼立即查到、且沒有決策價值的細節。
4. 未驗證且容易誤導未來任務的推測。

## 索引寫入位置

優先寫入最精準的位置：

| 資訊類型 | 優先位置 |
|---|---|
| 使用者穩定偏好 | `Memories/index/preferences.json` |
| 每次任務必讀壓縮索引 | `Memories/index/startup-index.json` |
| 高頻摘要快取 | `Memories/index/agent-profile.json` |
| 跨專案決策 | `Memories/index/decisions.jsonl` |
| 跨專案摘要 | `Memories/index/summaries.jsonl` |
| 單一專案事實 | `Memories/projects/<project-key>/facts.jsonl` |
| 單一專案決策 | `Memories/projects/<project-key>/decisions.jsonl` |
| 單一專案警告 | `Memories/projects/<project-key>/warnings.jsonl` |
| 單一專案 bug 修復 | `Memories/projects/<project-key>/bug-fixes.jsonl` |

若專案記憶目錄尚未建立，可先寫入 `index/*.jsonl`，但後續整理時應搬到 `projects/<project-key>/`。

## 檢索優先順序

依任務查詢記憶時，使用以下順序：

1. `Memories/index/startup-index.json`
2. `Memories/index/agent-profile.json`
3. 目前 repo 的 `AGENTS.md`
4. 目前專案的 `Memories/projects/<project-key>/*.jsonl`
5. `decision` 類記憶
6. `warning` 與 `bug_fix` 類記憶
7. `preference` 類記憶
8. `project_fact` 類記憶
9. `conversation` 類摘要
10. 必要時才依 `rawPath` 讀 raw memory

檢索時應先用關鍵字縮小範圍，不要整批讀入所有 JSONL。專案任務應優先使用專案記憶，再補全域偏好與決策。

同一批搜尋結果的排序規則：

1. 先保留 `active` 記憶；`superseded`、`archived` 與 `purge_candidate` 只在追溯來源或 active 記憶不足時讀。
2. 先依任務關鍵字、專案、類型與摘要內容判斷相關度。
3. 相關度接近、內容重疊或規則衝突時，越近期的改動越優先；日期欄位依序使用 `updatedAt`、`createdAt`、`timestamp`。
4. 若近期記憶是 `unverified`，不得直接覆蓋較舊但已驗證的 active 記憶；需要標示不確定或再查證。
5. 日期與相關度都接近時，再用 `importance` 輔助排序。

每次任務的固定記憶讀取應以 `startup-index.json` 為主；`preferences.json`、`agent-profile.json`、完整 JSONL 與 raw memory 都是按需補讀。

## Token 預算與成長控制

記憶系統必須防止長期使用後固定前置 token 無限擴大。

固定前置預算：

1. `startup-index.json` 目標低於 1200 tokens，硬上限 1800 tokens。
2. `AGENT_BOOTSTRAP.md` 只放不可跳過規則，目標低於 1500 tokens。
3. repo `AGENTS.md` 只放入口、路徑與專案特殊規則，避免放完整專案知識。
4. `Memories/logs/index.md` 啟動時只讀最新前 10 筆任務日誌，累積至 3000 tokens 為上限。
5. 每次任務從 JSONL 檢索出的摘要通常限制 5 到 10 筆；需要更多時應說明原因。
6. **前置總預算硬上限：5000 tokens**（合計 `AGENT_BOOTSTRAP.md` + repo `AGENTS.md` + `startup-index.json` + `logs/index.md` 前 10 筆）。超過時優先壓縮 `logs/index.md` 與 `startup-index.json`，不刪除核心規則。

若 `logs/index.md` 累積任務超過 100 筆，應建立封存檔（例如 `logs/archive-2026H1.md`），保留近期 50 筆在 active index。

active index 容量控制：

1. `startup-index.json` 只保存高頻且會影響每次任務的資訊；細節放回完整索引。
2. `preferences.json` 若超過約 30 筆 active preference，應合併同類偏好、標記 `superseded`，或轉移細節到 `summaries.jsonl`。
3. 單一 JSONL active 檔案若超過約 200 筆或 200 KB，應建立封存檔，例如 `summaries.archive-2026H1.jsonl`，並保留近期與高重要性摘要在 active 檔。
4. 專案 JSONL 若超過約 100 筆，應依 `facts`、`decisions`、`warnings`、`bug-fixes` 拆分，並封存低重要性或過期項目。
5. `raw` 可以長期保存，但不能成為固定讀取來源。
6. 任何新規則若需要每次任務都知道，優先濃縮進 `startup-index.json`，不要把完整長文加入必讀流程。

整理任務應優先做「合併、封存、標記狀態」，只有確認無追溯價值時才刪除。

## 重要性評分

每筆記憶應有 `importance`，範圍 `0.0` 到 `1.0`。

| 分數 | 意義 |
|---|---|
| `0.9 - 1.0` | 長期重要，會影響未來行為 |
| `0.7 - 0.89` | 專案或任務重要資訊，必須寫入正式索引 |
| `0.4 - 0.69` | 可參考，但不一定常用 |
| `0.0 - 0.39` | 低價值，通常不寫入正式索引 |

檢索排序建議公式：

```text
score = relevance * 0.45 + recency * 0.35 + importance * 0.2
```

其中 `relevance` 來自任務關鍵字、專案與類型匹配；`recency` 依 `updatedAt`、`createdAt`、`timestamp` 計算。若無法計算分數，人工檢索也應遵守「相關度接近時近期優先」。

## 記憶升級規則

當同一資訊多次出現，應從一般對話升級為穩定記憶。

範例：

1. 使用者多次要求繁體中文回答，寫入 `preferences.json`。
2. 某專案固定使用 Service 層處理商業邏輯，寫入 `projects/<project-key>/facts.jsonl`。
3. 某種 bug 重複發生，寫入 `projects/<project-key>/warnings.jsonl` 或 `bug-fixes.jsonl`。

## 記憶衝突規則

新記憶和舊記憶衝突時，不要直接刪掉舊記憶。應標記狀態。

```json
{
  "status": "superseded",
  "replacedBy": "new-memory-id",
  "supersededAt": "2026-06-22T10:00:00+08:00"
}
```

可用狀態：

| Status | 意義 |
|---|---|
| `active` | 目前有效 |
| `superseded` | 已被新記憶取代 |
| `unverified` | 尚未確認 |
| `archived` | 保留但不主動使用 |
| `purge_candidate` | 可在整理任務中評估刪除 |

## 淘汰與整理規則

第一版不自動刪除記憶，只標記狀態。整理任務可依以下規則處理：

1. `importance < 0.3` 且 180 天未引用：標記為 `archived`。
2. `importance < 0.2` 且 365 天未引用：標記為 `purge_candidate`。
3. `superseded` 記憶保留來源鏈結，不直接刪除。
4. `purge_candidate` 只有在確認無法提供追溯價值時才可刪除。

整理任務應記錄在 `Memories/logs/`。

## 任務日誌寫入

每次對話結束前，必須：

1. 在 `Memories/logs/` 創建日誌檔：`{任務名稱簡述}_{yyyyMMddHHmmss}.md`
2. 更新 `Memories/logs/index.md`：
   - 遞增總任務數
   - 在「最近任務」表格最上方新增一筆（最新在上）
   - 在「完整清單」節追加該筆記錄
   - 更新最後更新時間

日誌內容避免洩漏密碼、token、私鑰。摘要應具體可回溯，不寫「完成了一些修改」這類模糊敘述。

## Agent Profile

`Memories/index/agent-profile.json` 是高頻摘要快取，用來快速提供使用者偏好、常用專案與技術棧。

它不是唯一真實來源；若與 `preferences.json` 或專案記憶衝突，應以更具體、更新的來源為準。

日常啟動時優先使用 `startup-index.json`。`agent-profile.json` 可保留較多背景，但不應變成無上限必讀檔。

## 第一版 MVP

記憶系統第一版先採用檔案式設計：

1. `raw` 保存完整 JSON。
2. `index` 保存 JSONL 摘要。
3. `projects` 保存專案級 JSONL。
4. `preferences.json` 保存穩定偏好。
5. `agent-profile.json` 保存高頻摘要快取。
6. `startup-index.json` 保存每次任務必讀的壓縮啟動索引。
7. 不強制使用向量資料庫。
8. 後續資料量變大再加入 embedding 搜尋。
